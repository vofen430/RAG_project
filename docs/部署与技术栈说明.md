## 路线一：Chocolatey 自动安装（Windows）

> [Chocolatey](https://chocolatey.org/) 是 Windows 下的包管理器，类似 Linux 的 apt / macOS 的 brew。

### 1. 安装 Chocolatey

以**管理员权限**打开 PowerShell，运行：

```powershell
Set-ExecutionPolicy Bypass -Scope Process -Force
[System.Net.ServicePointManager]::SecurityProtocol = [System.Net.ServicePointManager]::SecurityProtocol -bor 3072
iex ((New-Object System.Net.WebClient).DownloadString('https://community.chocolatey.org/install.ps1'))
```

安装完成后关闭并重新打开 PowerShell，验证：

```powershell
choco --version
# 输出版本号即为成功
```

### 2. 一键安装所有依赖

```powershell
# 以管理员权限运行
choco install git docker-desktop openjdk17 nodejs-lts -y
```

> **说明：**
> - `docker-desktop` 包含 Docker Engine + Docker Compose
> - `openjdk17` 安装 JDK 17
> - `nodejs-lts` 安装最新 LTS 版 Node.js（含 npm）

### 3. 安装后重启终端，验证版本

```powershell
git --version            # git version 2.x.x
docker --version         # Docker version 2x.x.x
docker compose version   # Docker Compose version v2.x.x
java -version            # openjdk version "17.x.x"
node -v                  # v20.x.x 或更高
npm -v                   # 9.x.x 或更高
```

### 4. 后续升级

```powershell
# 升级全部已安装的包
choco upgrade all -y

# 升级单个包
choco upgrade openjdk17 -y
```

## 路线二：手动安装（Windows / macOS / Linux）

| 平台 | 安装方式 |
|------|---------|

| Linux (Ubuntu) | 参考 [Docker 官方指南](https://docs.docker.com/engine/install/ubuntu/) |

```bash
# Linux 快速安装 (Ubuntu)
curl -fsSL https://get.docker.com | sh
sudo usermod -aG docker $USER    # 免 sudo 运行 docker
newgrp docker                     # 立即生效（或重新登录）
```

### JDK 17

| 平台 | 安装方式 |
|------|---------|

## 基础设施启动（Docker Compose）

项目根目录下的 `docker-compose.yml` 定义了两个服务：

| 服务 | 镜像 | 端口 | 用途 |
|------|------|------|------|
| **rag-postgres** | `pgvector/pgvector:pg16` | 5432 | PostgreSQL 16 + pgvector 向量扩展 |
| **rag-redis** | `redis:7-alpine` | 6379 | Redis 7（JWT 黑名单 / 缓存） |

```bash
# 克隆项目
git clone https://github.com/vofen430/RAG_project.git
cd RAG_project

# 启动基础设施
docker compose up -d

# 验证
docker compose ps
# rag-postgres   running (healthy)
# rag-redis      running (healthy)
```

> 数据持久化在 Docker Volume 中（`pgdata`、`redisdata`），普通 `docker compose down` 不丢数据。
> 只有 `docker compose down -v` 才会清除。

## 后端启动

```bash
cd backend

# 设置环境变量
# Windows PowerShell:
#   $env:MODEL_PROVIDER_API_KEY = "sk-your-siliconflow-api-key"
#   $env:JWT_SECRET = "my-32-char-long-secure-secret-key"
# Linux / macOS / Git Bash:
export MODEL_PROVIDER_API_KEY="sk-your-siliconflow-api-key"
export JWT_SECRET="my-32-char-long-secure-secret-key"

# 启动（Maven Wrapper 会自动下载 Maven，首次约 2 分钟）
./mvnw spring-boot:run          # Linux / macOS / Git Bash
# 或
mvnw.cmd spring-boot:run        # Windows CMD
```

**启动成功标志：**
```
INFO  Migrating schema "public" to version "1 - init schema"
INFO  Migrating schema "public" to version "2 - seed default user"
INFO  Migrating schema "public" to version "3 - add api key to settings"
INFO  Started NovelRagApplication in X.XXX seconds
```

**环境变量说明：**

| 变量 | 必填 | 默认值 | 说明 |
|------|------|-------|------|
| `MODEL_PROVIDER_API_KEY` | ✅ | `your-api-key-here` | SiliconFlow API 密钥 |
| `JWT_SECRET` | ✅ | (见 application.yml) | JWT 签名密钥，至少 32 字符 |
| `SERVER_PORT` | ❌ | 8080 | 后端端口 |
| `DB_HOST` | ❌ | 127.0.0.1 | PostgreSQL 主机 |
| `DB_PORT` | ❌ | 5432 | PostgreSQL 端口 |
| `DB_NAME` | ❌ | rag_local | 数据库名 |
| `DB_USERNAME` | ❌ | rag_user | 数据库用户 |
| `DB_PASSWORD` | ❌ | rag_password | 数据库密码 |
| `REDIS_HOST` | ❌ | 127.0.0.1 | Redis 主机 |
| `REDIS_PORT` | ❌ | 6379 | Redis 端口 |
| `LOCAL_STORAGE_ROOT` | ❌ | ./storage | 文件存储根目录 |

## 常见问题

| 问题 | 原因 | 解决 |
|------|------|------|
| `JAVA_HOME not set` | JDK 未安装/未配置 | 安装 JDK 17 后重启终端 |
| 端口 5432 被占用 | 已有 PG 实例 | `docker compose down` → `docker compose up -d` |
| 端口 8080 被占用 | 其他服务 | `export SERVER_PORT=8081` |
| `npm install` 报错 | Node 版本低 | 升级到 20+：`nvm install --lts` |
| API 调用 401 | API Key 无效 | 通过设置页面或环境变量重新配置 |
| 文件上传找不到 | 存储路径 | 检查 `LOCAL_STORAGE_ROOT` |

## 按 RAG 流程分步讲解每个库的作用

### 第 0 步：项目启动与配置

| 库 | Maven 坐标 | 版本 | 作用 |
|----|-----------|------|------|
| **Spring Boot** | `spring-boot-starter-parent` | 3.2.5 | 整个后端框架，提供自动配置、依赖管理（BOM）、内嵌 Tomcat 服务器 |
| **Lombok** | `org.projectlombok:lombok` | BOM 管理 | 简化 Java 代码（注解生成 getter/setter），编译时生效，不进入运行时 |
| **Jackson** | `jackson-databind` | BOM 管理 | JSON 序列化/反序列化，处理所有 HTTP 请求体和响应体的 JSON 转换 |
| **Spring Actuator** | `spring-boot-starter-actuator` | BOM 管理 | 提供 `/actuator/health` 健康检查端点，用于监控和部署验证 |

> **启动流程：** `NovelRagApplication.main()` → Spring Boot 自动扫描 → 创建所有 Bean → 连接 PG/Redis → 执行 Flyway 迁移 → 监听端口 8080

---

### 第 1 步：用户认证（登录 → JWT 令牌）

| 库 | Maven 坐标 | 版本 | 作用 |
|----|-----------|------|------|
| **Spring Security** | `spring-boot-starter-security` | BOM 管理 | 安全框架，拦截所有 HTTP 请求进行认证/授权检查 |
| **JJWT** | `io.jsonwebtoken:jjwt-api` / `jjwt-impl` / `jjwt-jackson` | 0.12.6 | JWT（JSON Web Token）的生成、签名和验证 |
| **Spring Data Redis** | `spring-boot-starter-data-redis` | BOM 管理 | 操作 Redis，用于存储 JWT 刷新令牌和黑名单 |
| **Spring Validation** | `spring-boot-starter-validation` | BOM 管理 | `@NotBlank` `@NotNull` 等注解验证请求参数的合法性 |

**流程：**
```
POST /api/auth/login (username, password)
  → Spring Security 验证密码
  → JJWT 生成 accessToken (2小时) + refreshToken
  → refreshToken 存入 Redis
  → 返回 JWT 给前端

后续请求 → JwtAuthenticationFilter 拦截
  → 从 Authorization: Bearer <token> 提取 JWT
  → JJWT 验证签名和过期时间
  → 通过则放行，否则返回 401
```

---

### 第 2 步：文档上传（接收文件 → 存储到磁盘 → 写入数据库）

| 库 | Maven 坐标 | 版本 | 作用 |
|----|-----------|------|------|
| **Spring Web** | `spring-boot-starter-web` | BOM 管理 | 提供 `MultipartFile` 接收上传文件、REST Controller、内嵌 Tomcat |
| **MyBatis-Plus** | `mybatis-plus-spring-boot3-starter` | 3.5.7 | ORM 框架，自动生成 SQL，将 Java 对象映射到数据库表 |
| **PostgreSQL Driver** | `org.postgresql:postgresql` | BOM 管理 | JDBC 驱动，让 Java 能连接 PostgreSQL 数据库 |
| **Flyway** | `org.flywaydb:flyway-core` | BOM 管理 | 数据库版本迁移工具，自动执行 V1/V2/V3 SQL 建表和种子数据 |

**流程：**
```
POST /api/documents (multipart file)
  → DocumentController 接收 MultipartFile
  → DocumentService:
      1. 检测编码 (UTF-8 / GBK)
      2. 计算 SHA-256 哈希
      3. 写入磁盘: storage/documents/{userId}/{docId}.txt
      4. MyBatis-Plus INSERT → documents 表
  → 返回文档元数据 JSON
```

**Flyway 迁移文件：**
```
V1__init_schema.sql          → 创建 9 张表 + pgvector 扩展
V2__seed_default_user.sql    → 插入 demo 用户
V3__add_api_key_to_settings.sql → 添加 api_key 列
```

---

### 第 3 步：文本分块（Chunking）

| 库 | 版本 | 作用 |
|----|------|------|
| **Java 标准库** | JDK 17 | 正则表达式 (`Pattern`)、字符串操作、UUID 生成 |

> 此步骤不依赖第三方库，完全用 Java 标准库实现。

**流程：**
```
ChunkingService.chunkDocument(text, chunkSize=500, chunkOverlap=100)
  → 按"第X幕"正则分割为结构化章节
  → 每个章节内按固定窗口滑动分块（500字符，100字符重叠）
  → 提取人物实体（周朴园、繁漪、鲁侍萍...）
  → 生成 DocumentChunkEntity 列表（含 chunkIndex, sectionLabel, contentText）
  → 输出: 雷雨.txt → 227 个分块
```

---

### 第 4 步：向量嵌入（Embedding）

| 库 | Maven 坐标 | 版本 | 作用 |
|----|-----------|------|------|
| **Spring WebFlux** | `spring-boot-starter-webflux` | BOM 管理 | 提供 `WebClient` 异步 HTTP 客户端，调用 SiliconFlow API |
| **Jackson** | `jackson-databind` | BOM 管理 | 将 API 请求/响应 JSON 与 Java Map 互转 |

**流程：**
```
EmbeddingService.embedBatch(texts, model="Qwen/Qwen3-Embedding-0.6B")
  → WebClient POST https://api.siliconflow.com/v1/embeddings
    请求体: {"model":"Qwen/Qwen3-Embedding-0.6B", "input":["文本1","文本2",...]}
  → 返回 1024 维浮点数向量
  → 向量写入 DocumentChunkEntity.embeddingVector
```

---

### 第 5 步：向量持久化（Indexing → pgvector）

| 库 | Maven 坐标 | 版本 | 作用 |
|----|-----------|------|------|
| **pgvector Java** | `com.pgvector:pgvector` | 0.1.6 | 将 Java `float[]` 转换为 PostgreSQL `vector` 类型，支持余弦相似度检索 |
| **MyBatis-Plus** | `mybatis-plus-spring-boot3-starter` | 3.5.7 | 批量 INSERT 分块记录到 `document_chunks` 表 |

**流程：**
```
RagPipelineService.indexDocument()
  → MyBatis-Plus 批量 INSERT document_chunks
  → 每条记录含:
      content_text (原文文本)
      embedding_vector (1024维向量，pgvector 类型)
      section_label, chunk_index, start_offset, end_offset
```

**pgvector 扩展的作用：**
```sql
-- PostgreSQL 中的向量列定义
embedding_vector VECTOR(1024)

-- 余弦相似度检索（由 RetrievalService 使用）
SELECT * FROM document_chunks
ORDER BY embedding_vector <=> '[0.1, 0.2, ...]'::vector
LIMIT 10;
```

---

### 第 6 步：用户提问 → 查询嵌入

复用 **第 4 步** 的 `EmbeddingService`，将用户的查询文本转为同维度（1024维）向量。

---

### 第 7 步：Top-K 向量检索

| 库 | 作用 |
|----|------|
| **pgvector** | PostgreSQL `<=>` 余弦距离运算符，返回最相似的 K 条分块 |
| **MyBatis-Plus** | 执行自定义 SQL 查询 |

**流程：**
```
RetrievalService.retrieve(queryVector, topK=10)
  → SQL: SELECT ... FROM document_chunks ORDER BY embedding_vector <=> ? LIMIT 10
  → 返回 10 条最相似的分块（含原文和元数据）
```

---

### 第 8 步：语义重排（Reranking）

| 库 | 作用 |
|----|------|
| **Spring WebFlux / WebClient** | 调用 SiliconFlow Rerank API |
| **Jackson** | JSON 序列化 |

**流程：**
```
RerankService.rerank(query, documents, model="Qwen/Qwen3-Reranker-0.6B", topN=5)
  → WebClient POST https://api.siliconflow.com/v1/rerank
    请求体: {"model":"...", "query":"用户问题", "documents":["分块1","分块2",...], "top_n":5}
  → 返回按相关性重排的结果 + 精排分数（0~1）
  → 从 10 条缩减到 5 条最相关的
```

---

### 第 9 步：Prompt 构建 + LLM 流式生成

| 库 | 作用 |
|----|------|
| **Spring WebFlux** | `WebClient` 发起 SSE（Server-Sent Events）流式请求，逐 token 接收 |
| **Spring Web** | `Flux<String>` 返回 SSE 流给前端 |
| **Jackson** | 解析流式 JSON 响应 |

**流程：**
```
ChatService.streamChat(messages, model="Qwen/Qwen3-8B")
  → 构建 system prompt（含检索到的证据分块原文）
  → WebClient POST https://api.siliconflow.com/v1/chat/completions
    请求体: {"model":"...", "messages":[...], "stream":true}
  → 接收 SSE 流: data: {"choices":[{"delta":{"content":"周"}}]}
  → 逐 token 转发给前端: event: token\ndata: {"content":"周","traceId":"xxx"}
```

---

### 第 10 步：Trace 持久化（证据溯源）

| 库 | 作用 |
|----|------|
| **MyBatis-Plus** | 将 Trace（查询记录）和 Evidence（证据项）写入数据库 |

**流程：**
```
RagPipelineService.query()
  → 创建 RagQueryTraceEntity (用户问题, 延迟, 检索参数)
  → 创建 TraceEvidenceItemEntity × N (每条证据的分块ID, 检索分数, rerank分数)
  → MyBatis-Plus INSERT → rag_query_traces + trace_evidence_items 表
  → 前端可通过 GET /api/chat/traces/{traceId} 查看完整证据链
```

| 服务 | 提供商 | 用途 | 使用的 API |
