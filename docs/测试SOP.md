# 测试标准操作规程（SOP）

> 本文档涵盖 **Novel RAG 人物关系分析系统** 的快速环境搭建流程和功能测试流程。

---

## 一、快速环境搭建流程

### 1.1 前置条件

| 工具 | 最低版本 | 安装方式（Windows） |
|------|---------|-------------------|
| Git | 任意 | `choco install git` |
| JDK | 17 | `choco install openjdk` |
| Node.js | 24 | `choco install nodejs-lts` |
| npm | 11 | 随 Node.js 自动安装 |
| Maven | 无需安装 | 项目自带 Maven Wrapper（`mvnw.cmd`） |

> 如果你的系统已安装 Chocolatey，可一行命令安装全部工具：
> ```bash
> choco install git openjdk nodejs-lts -y
> ```

### 1.2 搭建步骤

```
步骤 1：克隆项目
       → git clone https://github.com/vofen430/RAG_project.git
       → cd RAG_project

步骤 2：启动后端
       → cd backend
       → ./mvnw.cmd spring-boot:run
       → 等待控制台出现 "Started NovelRagApplication" 字样
       → 后端运行在 http://localhost:8080

步骤 3：启动前端（新开一个终端窗口）
       → cd frontend
       → npm install
       → npm run dev
       → 前端运行在 http://localhost:5173

步骤 4：浏览器验证
       → 打开 http://localhost:5173
       → 确认能看到深色主题的 "Novel RAG" 界面
```

### 1.3 环境搭建验证清单

完成搭建后，逐项检查以下内容：

| 序号 | 检查项 | 预期结果 | 通过 |
|------|--------|---------|------|
| 1 | 后端编译 | `./mvnw.cmd clean compile` 输出 BUILD SUCCESS | ☐ |
| 2 | 前端构建 | `npm run build` 输出 ✓ built 无报错 | ☐ |
| 3 | 后端启动 | 控制台输出 "Started NovelRagApplication"，端口 8080 | ☐ |
| 4 | 前端启动 | 控制台输出 "VITE ready"，端口 5173 | ☐ |
| 5 | 页面加载 | 浏览器访问 localhost:5173 显示深色主题界面 | ☐ |
| 6 | API 连通 | 设置页面能加载模型下拉列表（从后端获取 models.json 数据） | ☐ |

### 1.4 常见搭建问题

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| `mvnw.cmd` 报错 "JAVA_HOME not set" | JDK 未安装或环境变量未生效 | 安装 JDK 后重启终端，或手动设置 `$env:JAVA_HOME` |
| `npm install` 报错 "engine not compatible" | Node.js 版本低于 24 | 升级 Node.js：`choco upgrade nodejs-lts` |
| 前端页面空白 | 后端未启动，API 请求失败 | 先启动后端，再刷新前端页面 |
| 端口 8080 被占用 | 其他服务占用了端口 | 终止占用进程：`netstat -ano | findstr :8080` 然后 `taskkill /PID <PID> /F` |

## 二、功能测试流程

### 2.1 测试前准备

```
准备 1：确保后端和前端均已启动（参见第一章）
准备 2：准备一个 SiliconFlow API Key（从 https://cloud.siliconflow.com/account/ak 获取）
准备 3：准备测试用小说文件（项目自带 雷雨.txt，可直接使用）
```

### 2.2 功能测试项

#### 测试 1：API 密钥配置

| 步骤 | 操作 | 预期结果 |
|------|------|---------|
| 1.1 | 浏览器打开 `http://localhost:5173/settings` | 显示模型设置页面 |

| 1.3 | 点击 "保存密钥" 按钮 | 页面显示 "✅ 设置已保存" |

#### 测试 2：模型切换

| 步骤 | 操作 | 预期结果 |
|------|------|---------|
| 2.1 | 在设置页面查看 "嵌入模型" 下拉框 | 默认选中 "Qwen3 Embedding 0.6B (Free)" |
| 2.2 | 切换嵌入模型到 "Qwen3 Embedding 4B" | 下拉框更新为新选项 |
| 2.3 | 查看 "重排序模型" 下拉框 | 显示 3 个 Qwen3 Reranker 选项 |
| 2.4 | 查看 "生成模型" 下拉框 | 显示 6 个 LLM 选项（Qwen3/DeepSeek 系列） |
| 2.5 | 勾选任意模型的 "自定义模型名称" 复选框 | 出现文本输入框，可输入自定义模型名 |
| 2.6 | 在自定义输入框中输入 `BAAI/bge-m3` | 输入框接受自定义模型名 |
| 2.7 | 点击 "💾 保存所有设置" | 显示 "✅ 设置已保存" |

#### 测试 3：文档上传

| 步骤 | 操作 | 预期结果 |
|------|------|---------|
| 3.1 | 导航到 `http://localhost:5173/upload` | 显示文档管理页面，有拖拽上传区域 |
| 3.2 | 点击上传区域，选择 `雷雨.txt` 文件 | 显示上传进度，完成后文件出现在文档列表中 |
| 3.3 | 查看上传后的文档状态 | 显示 "已上传"，文件名和大小正确 |
| 3.4 | 点击 "开始索引" 按钮 | 状态依次变为 "分块中..." → "嵌入中..." → "已索引 ✓" |
| 3.5 | 等待索引完成 | 显示分块数量（预期约 150-200 个分块），状态为绿色 "已索引 ✓" |

> **⚠️ 注意：** 索引过程需要调用 SiliconFlow Embedding API，耗时取决于文本长度和网络速度，通常需要 30 秒至 2 分钟。

### 2.3 完整 RAG 管线验证

以下流程验证 10 步 RAG 管线的完整性：

```
 Ingestion（文档摄入）
   ↓  → 验证方法：上传文件后，GET /api/documents 返回文档信息
 Chunking（文本分块）
   ↓  → 验证方法：索引时状态变为 "CHUNKING"，totalChunks > 0
 Embedding（向量嵌入）
   ↓  → 验证方法：状态变为 "EMBEDDING"，processedChunks 逐步增加
 Indexing（向量索引）
   ↓  → 验证方法：状态变为 "INDEXED"
 User Query（用户提问）
   ↓  → 验证方法：POST /api/chat/stream 发送问题
 Query Embedding（查询嵌入）
   ↓  → 验证方法：后端日志输出 "Embedded batch..."
 Top-K Retrieval（检索）
   ↓  → 验证方法：后端日志输出 "Retrieved X chunks"
 Reranking（重排序）
   ↓  → 验证方法：后端日志输出 "Reranked X chunks to Y"
 Prompt Construction（提示构建）
   ↓  → 验证方法：后端日志输出 "Constructed prompt with X context chunks"
 LLM Generation（大模型生成）
   ↓  → 验证方法：前端逐字流式显示回答内容
```

## 三、自动化验证命令

以下命令可在 CI/CD 环境或本地快速验证项目可用性：

```bash
# 后端编译验证
cd backend
./mvnw.cmd clean compile -q
echo "后端编译: 通过"

# 前端构建验证
cd ../frontend
npm install --silent
npm run build
echo "前端构建: 通过"
```

## 四、测试环境快速重置

如需清除所有运行数据，执行以下操作：

```bash
# 停止服务
# → 在终端中按 Ctrl+C 停止后端和前端进程

# 清理后端编译产物
cd backend
./mvnw.cmd clean

# 清理前端构建产物
cd ../frontend
rm -rf dist/

# 重新启动（参见第一章 1.2 节）
```

> **提示：** 由于本系统使用内存向量存储，重启后端后所有已索引的文档数据会被清除，需要重新上传和索引。
