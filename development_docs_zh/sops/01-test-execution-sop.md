# 测试执行 SOP

## 1. 目的

本 SOP 定义本地 RAG 项目的标准测试执行流程。

## 2. 前置条件

开始测试前，确认：

- PostgreSQL 已在本地运行
- Redis 已在本地运行
- 所需环境变量已配置
- 测试数据库可用
- 后端依赖已安装
- 前端依赖已安装

## 3. 执行顺序

按以下顺序执行测试：

1. 后端单元测试
2. 后端集成测试
3. 前端单元测试
4. 端到端测试
5. 手工功能验证

## 4. 后端单元测试 SOP

步骤：

1. 打开项目根目录。
2. 进入 `backend` 目录。
3. 运行后端单元测试命令。
4. 保存测试结果输出。
5. 若核心模块测试失败，立即停止。

## 5. 后端集成测试 SOP

步骤：

1. 确认 PostgreSQL 和 Redis 可用。
2. 确认数据库 schema 已初始化。
3. 运行后端集成测试命令。
4. 保存测试结果输出。
5. 记录失败 case 的模块名和错误信息。

## 6. 前端单元测试 SOP

步骤：

1. 进入 `frontend` 目录。
2. 运行前端单元测试命令。
3. 保存测试结果输出。
4. 记录失败 case 的页面或组件名。

## 7. 端到端测试 SOP

步骤：

1. 启动 PostgreSQL 和 Redis。
2. 启动后端。
3. 启动前端。
4. 运行端到端测试。
5. 仅在失败时保存截图或日志。
6. 记录失败 case 的路由和操作名。

## 8. 手工功能验证 SOP

按顺序执行以下检查清单：

1. 使用有效账户登录。
2. 上传一个文档。
3. 启动索引。
4. 等待索引完成。
5. 创建一个聊天会话。
6. 提交一个问题。
7. 等待答案流式输出完成。
8. 打开 trace 详情。
9. 打开至少一个 citation 条目。
10. 更新用户设置。
11. 提交答案反馈。

## 9. 缺陷记录 SOP

每个失败测试都要记录：

- 测试级别
- 模块名
- 精确操作
- 输入数据
- 实际结果
- 期望结果
- 错误日志
- 是否可复现

## 10. 退出标准

只有以下条件满足时才可关闭测试：

- 所有关键测试通过
- 没有阻塞性缺陷处于打开状态
- 手工功能验证完成
- 验收清单已签字
